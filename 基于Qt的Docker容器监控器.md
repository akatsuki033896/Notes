# 介绍

我做了一个Docker 容器监控工具，该容器监控工具面向Windows，Linux，macOS等多种系统的宿主机，核心是通过 Docker API 获取容器的运行状态和进行资源监控。基于Qt的GUI界面进行实时展示，GUI的信息每秒更新一次，实现近实时监控。

---

我做了一个基于 Qt 的 Docker 容器监控工具，支持 Windows、Linux 和 macOS，主要通过 Docker API 获取容器的运行状态，并在界面上进行实时展示。在架构上，我把系统分成数据获取和界面展示两部分，底层通过命令或 API 获取容器的 CPU、内存等信息，并解析 JSON 数据存入结构体，上层用 Qt 做实时可视化。在实现上，我主要做了两点优化：一是通过 QTimer 配合事件循环来做定时刷新，避免频繁创建线程，同时保证 UI 不阻塞；二是引入线程池处理数据获取任务，这样可以同时监控多个容器的资源使用情况。封装了一个 Container Manager 来管理容器数据，方便后续扩展，比如增加更细粒度的监控功能。

## 技术要点

- 解析 Docker 返回的 JSON 数据并存入结构体，利用开源的 JSON 解析库
- 基于 qt 的 `QTimer` 定时器辅助线程池，同时把容器监控器优化成基于事件循环。使用多线程后可以同时监控大量容器的CPU占用，内存占用，查看端口映射、挂载地址等
- `QTimer` 基于事件循环，不需要操作系统分配独立的线程资源，避免频繁创建销毁线程的开销，适合更新界面这种短小的任务，避免线程阻塞，同时避免直接在程序入口加锁解锁操作，把锁封装在容器管理器这个底层，`QTimer` 很方便的修改几秒刷新一次 UI

## 解决问题

1. Windows平台下由于系统特性，在程序执行docker命令的时候会不停的弹出命令行窗口，GUI的窗口还会无法拖动。原本使用的是管道通信等api `p_open`，这个函数在win上会持续调用命令行导致 UI 问题。之后为了跨平台的优化，使用 `QObject`，设置信号槽来通信，例如每调用一次 `docker inspect` 或 `docker stat` 就发射信号，主窗口获取信号来更新UI，就防止了卡顿
2. Qt主要用QWidget，QSS的样式设计我使用chatgpt、opencode等工具进行辅助
3. 优化了代码的可扩展性，封装 `Container Manager`，利于之后添加功能例如调用 `top` 监视容器内线程情况
4. 开发和测试都由我自己完成，测试使用了 `nginx` 的容器和 `busybox`，模拟真实工业环境下上位机监控各个子系统的环境。

## 收获

这个项目其实也让我对上位机监控类软件有了一些理解，比如如何处理实时数据和界面响应，这一点和工业监控系统比较类似

## 为什么不用多线程刷新 UI

Qt 的 UI 必须在主线程更新，所以我把数据获取放在子线程，通过信号槽通知主线程更新界面

## 怎么避免 UI 卡顿

数据获取不在主线程，信号槽异步更新。

## 容器太多

使用线程池并发获取，同时可以限制线程数量，避免资源占用过高
